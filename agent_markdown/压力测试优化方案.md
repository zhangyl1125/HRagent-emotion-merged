# HRagent-05 压力测试瓶颈分析与可执行优化方案

## 结论

本次 Locust 页面显示：

- 总请求数：14
- 总失败数：0
- 失败率：0%
- 聚合平均耗时：3908.68 ms
- 聚合 P95：29000 ms
- 完整链路接口均已出现，包括登录、创建会话、员工查询、文档解析、Profile/Intent/Persona 设置、谈前指导、预演消息、结束预演、复盘报告、会话读取。

因此，本次测试结果可以判定为：**完整业务流程已跑通，且本轮无接口失败**。

但这不是正式高并发压测结论。因为每个核心接口仅执行 1 次，更适合作为 **1 用户完整流程冒烟测试**。从耗时看，系统主要瓶颈集中在 LLM 生成、报告生成、预演回复和文档解析。

## 当前主要瓶颈

| 接口 | 当前耗时 | 风险等级 | 说明 |
| --- | ---: | --- | --- |
| `POST /api/v1/guidance/[session]/stream` | 29409.81 ms | 高 | 谈前指导生成约 29.4 秒，是当前最慢接口 |
| `POST /api/v1/reports/[session]/coach` | 12649.88 ms | 高 | 复盘报告生成约 12.6 秒 |
| `POST /api/v1/rehearsal/[session]/message` | 7156.66 ms | 中高 | 预演员工回复约 7.1 秒 |
| `POST /api/v1/documents/text` | 4906.71 ms | 中 | 文档文本解析约 4.9 秒 |
| 其他 setup / session / employee 接口 | 26-150 ms | 低 | 基本正常 |

## 优化目标

1. 保持现有业务逻辑不变。
2. 优先降低用户感知等待时间。
3. 对可复用结果接入缓存。
4. 对长耗时任务改为异步或流式可见。
5. 后续正式高并发前，先完成单用户链路稳定性与缓存命中验证。

## 可执行方案

### 1. 谈前指导 `guidance` 优化

现状：

- `POST /api/v1/guidance/[session]/stream` 耗时约 29.4 秒。
- 该接口虽然是 stream，但整体完成时间仍较长。

执行方案：

1. 检查 `backend/services/guidance_service.py` 是否已优先读取 Redis guidance cache。
2. 缓存 key 建议包含：
   - employee profile hash
   - intent id
   - persona id
   - difficulty id
   - guidance prompt version
3. 命中缓存时直接返回已生成 guidance，不再调用 LLM。
4. 未命中时保留流式输出，但前端应尽早展示 section 级内容。
5. 将 guidance 生成拆成固定结构：
   - 沟通目标
   - 开场建议
   - 风险提醒
   - 应对策略
   - 推荐话术
6. 后端按 section 增量写入缓存，避免中途失败后完全重来。

验收标准：

- 第二次相同 session 输入生成 guidance，应从约 29 秒降到 1 秒以内。
- 前端 2 秒内能看到第一段谈前指导内容。

### 2. 复盘报告 `coach report` 优化

现状：

- `POST /api/v1/reports/[session]/coach` 耗时约 12.6 秒。

执行方案：

1. 检查 `backend/services/coach_service.py` 是否按 session_id 缓存最终 coach report。
2. 报告生成改成“先查缓存，再生成”的模式：
   - 若 `coach_reports` 表已有记录，直接返回。
   - 若 Redis 有 report draft，也优先返回。
3. 将报告拆为并行任务：
   - Rubric 评估
   - 情绪承接评估
   - 绩效反馈质量评估
   - 话术红线检测
4. 所有子任务完成后再合成最终报告。
5. 对规则型红线检测优先本地执行，不走 LLM。

验收标准：

- 已生成报告再次请求时小于 1 秒。
- 首次生成目标控制在 6-8 秒以内。
- Locust 中 `POST /api/v1/reports/[session]/coach` 不再出现 status=0 或超时。

### 3. 预演消息 `rehearsal message` 优化

现状：

- `POST /api/v1/rehearsal/[session]/message` 耗时约 7.1 秒。

执行方案：

1. 保持完整 AI 回复不做长期缓存，避免多轮对话不自然。
2. 只缓存中间结果：
   - persona prompt
   - dynamic emotion state
   - motivation score
   - conversation summary
3. 预演回复接口启用流式返回，让用户先看到“员工正在回应”的文字片段。
4. 若 TTS 开启，文字先返回，语音异步生成，不阻塞文本展示。
5. 对动态情绪计算使用本地规则优先，LLM 只处理必要的语义补充。

验收标准：

- 文本首 token / 首段展示小于 2 秒。
- 完整回复目标控制在 4-6 秒以内。
- TTS 失败不影响文本回复。

### 4. 文档解析 `documents/text` 优化

现状：

- `POST /api/v1/documents/text` 耗时约 4.9 秒。

执行方案：

1. 对上传文本计算 hash。
2. 以 `document_parse:{hash}:{profile_prompt_version}` 作为 Redis cache key。
3. 相同文本再次解析时直接返回缓存结果。
4. 对纯 JSON / 结构化文本优先走本地解析，不调用 LLM。
5. LLM 提取失败时，优先使用已有 JSON 字段兜底补齐。

验收标准：

- 相同文档第二次解析小于 1 秒。
- JSON 格式测试文档不再依赖 LLM。
- 不再出现 `employee_alias is missing` 类结构化字段缺失错误。

第一阶段：1 用户完整流程

- 目标：观察 LLM 接口是否排队、是否出现 502/504/status=0。

第三阶段：20 用户业务并发

- Users: 20

- 目标：验证 Redis、PostgreSQL、后端 worker 是否稳定。

第四阶段：100 用户登录与轻量读接口

- Users: 100
- Ramp up: 5
- 目标：验证认证、多 session、Redis active session 上限。
- 不建议直接用 100 用户跑完整 LLM 六步流程，容易对模型服务造成压力。

## 优先级

1. 先做 Guidance 缓存与 section 级流式展示。
2. 再做 Coach Report 缓存与子任务并行。
3. 然后做 Rehearsal 中间结果缓存和文本优先返回。
4. 最后做 Documents/Text hash 缓存与 JSON 本地解析。

## 风险说明

- 当前 0% 失败只代表 1 用户完整链路成功，不代表 100 用户完整链路可稳定运行。
- LLM 类接口本身耗时长，直接扩大 Users 会快速放大排队和超时。
- 正式高并发前，应先确认缓存命中、超时设置、后端 worker 数量和模型服务限流策略。
