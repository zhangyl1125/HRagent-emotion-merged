# 阶段 04：缓存、Singleflight、重试与幂等

## 目标

消除跨 worker 的重复重计算和客户端重放，限制重试放大，同时保持动态员工回复不被缓存。

## 现状

- Guidance 与 CoachReport 已有缓存和部分进程内锁，但缓存未命中时多个 worker 仍可能重复生成。
- 通用重试缺少错误分类、指数退避、抖动和 `Retry-After` 支持。
- 网络重放可能重复追加经理消息或重复启动重任务。

## 改动

1. Guidance 和 CoachReport 按语义缓存键增加 Redis singleflight；leader 生成，followers 等待同一缓存结果。
2. leader 失败时只允许一个 follower 接替，其他请求继续有界等待，避免惊群。
3. 所有缓存键包含 provider、model、Prompt 版本、知识库版本和影响输出的业务输入。
4. 仅对连接错误、429、502、503、504 重试，采用指数退避加抖动并尊重 `Retry-After`。
5. 流式响应输出首个业务 delta 后不得自动重放模型请求。
6. Guidance、Report 和预演消息支持可选 `Idempotency-Key`；预演重复请求从 SessionState 已保存 turn 重建响应，不在 Redis 保存完整员工回复。

## 接口与配置

- 新增可选请求头 `Idempotency-Key`，未携带时保持现有行为。
- 同一用户、接口和 key 的请求体摘要不一致时返回 409 `idempotency_conflict`。
- 新增非敏感重试参数：最大次数、基础退避、最大退避和抖动比例。
- 不改变成功响应结构和 SSE 事件。

## 测试

- 多 worker 同时请求相同 Guidance/Report，确认只发生一次模型生成。
- 模拟 leader 崩溃、锁超时和缓存写失败，确认只有一个接替者。
- 对同一预演 `Idempotency-Key` 重放，确认会话只增加一对 turn。
- 模拟不可重试 400、可重试 429/503、`Retry-After` 和流式半途断开。
- 扫描 Redis 值，确认没有完整员工回复或敏感正文副本。

## 验收

- 相同语义重请求的模型调用数为 1。
- 重试次数和退避符合配置，不出现同步重试风暴。
- 幂等重放不重复修改会话状态。
- 完整员工回复只存在于业务会话存储，不进入通用缓存。

## 回滚

分别关闭 singleflight、幂等和新重试策略开关，恢复阶段 03 行为；保留已写缓存直到原 TTL 到期，不批量删除。

## 停止条件

需要改变成功响应、保存完整员工回复、删除现有缓存或修改数据库结构时停止并重新确认。
