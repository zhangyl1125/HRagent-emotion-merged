# 阶段 07：发布、回滚与持续优化

## 目标

将已通过 50 并发验收的应用层优化分批启用，保证可独立回滚，并建立持续 Token 与容量治理。

## 现状

- 本专项不修改模型、外部服务地址、Docker、GPU 或数据库结构。
- Token 压缩、并发协调和 singleflight 的风险不同，必须能够分别关闭。
- 外部 API 配额和响应时间会变化，需要持续观察而不能只依赖一次压测。

## 改动

1. 提供独立功能开关：上下文压缩、调用合并、并发协调、singleflight 和幂等。
2. 按 5%、25%、50%、100% 流量逐级启用，每级至少观察一个完整业务周期。
3. 每级比较 Token/流程、完整率、队列等待、429/5xx、质量门禁和缓存命中。
4. 建立每日 Token 预算、容量错误和队列等待检查；阈值超限时停止扩量。
5. 记录配置版本、发布时间、观察结论和回滚结果，不记录业务正文。

## 接口与配置

- 新增的功能开关默认关闭，只有阶段 06 证据通过后才逐级启用。
- 所有外部 API Provider、模型和地址保持现状；变更它们必须创建独立方案并重新授权。
- 不合并或移植 `ollama` 分支，不操作 Ollama 容器。

## 测试

- 在发布前后执行 1 用户完整冒烟和关键单测。
- 对每个开关单独验证关闭后恢复上一阶段行为。
- 模拟 Redis 故障、外部 API 429、SSE 断开和 worker 重启，验证回滚路径。
- 发布后核对 Grafana、数据库用量和服务健康。

## 验收

- 100% 启用后连续一个观察周期满足阶段 05、06 全部门禁。
- 任一功能可独立关闭，回滚不需要数据库清理或服务地址切换。
- 输入 Token 降幅保持至少 50%，50 用户目标无回归。
- 无敏感日志、无未授权外部系统变更、无 Ollama 依赖。

## 回滚

按幂等、singleflight、并发协调、调用合并、上下文压缩的逆序关闭开关；重建后端仅在配置不能热加载时执行。保留观测数据用于复盘。

## 停止条件

任一级出现完整率、质量、Token、队列或安全回归时停止扩量并回滚；需要模型路由、服务扩容或基础设施变更时另立 Spec 并重新授权。
