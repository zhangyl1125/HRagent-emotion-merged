# 验证、发布与回滚

## 实施前质量门禁

定义并记录：

- 最少请求数、最少完整流程数和最大失败率。
- Schema 首次解析成功率。
- 流式事件名称、顺序、错误和完成标记。
- 关键事实、引用、安全规则和个性化保留率。
- 相对黄金样本允许的质量漂移。
- 最大重试放大、限流、回退和队列等待。
- 完整业务结果的 Token 与费用目标。

拒绝零请求测试通过。分别统计已开始、已完成和失败流程。

## 对比要求

优化前后必须保持：

- 相同合成输入和业务流程。
- 相同测试规模、模型参数和时间口径。
- 相同价格版本、币种和费用公式。
- 相同质量、Schema、安全和流式检查。
- 明确的 Provider 返回、估算和不可用用量来源。

不能把缓存预热、异常重试或测试账号差异隐藏在汇总结果中。

## 必测场景

- 正常完整流程。
- 边界长度、空值、多语言和异常输入。
- Provider 限流、超时、临时错误和永久错误。
- 缓存命中、未命中、过期和隔离。
- Singleflight 所有者成功、失败和超时。
- 功能开关关闭后的旧路径。
- 流式中断、重连与幂等。
- 上下文压缩前后的关键事实和引用保留。

## 发布步骤

1. 完成离线黄金样本对比。
2. 在测试环境执行固定完整流程。
3. 使用影子模式或小流量启用新路径。
4. 观察费用、质量、延迟、重试、缓存和队列指标。
5. 达到预设观察期后逐步扩大范围。
6. 保留旧路径，直至生产证据稳定。

## 回滚条件

出现以下任一情况立即关闭新路径：

- 安全、权限、隐私或租户隔离失败。
- Schema、流式事件或成功响应契约变化。
- 关键事实、证据或业务质量超过容许回归。
- 完整流程费用上升或重试放大超过门禁。
- 队列持续饱和、失败率或 P95 延迟超过阈值。
- 用量数据失真，无法判断实际成本。

## 成功判定

只有同时满足以下条件才能宣布成功：

- 优化前后运行相同且非零的完整流程。
- Provider 用量或可靠事件与内部聚合一致。
- 完整业务结果费用下降。
- 所有质量、安全、Schema 和流式门禁通过。
- 回滚路径经过验证。
