# 计量、费用与基线

## 计量单位

按完整且成功的用户可见业务结果计量。单次调用变短不代表完整流程更便宜；重试、修复调用、缓存失效和路由升级都可能放大总费用。

至少记录：

| 字段 | 用途 |
| --- | --- |
| 流程、任务、功能 | 将用量归因到业务结果 |
| Provider、模型、版本 | 关联能力、价格和行为变化 |
| 输入、输出、推理、缓存 Token | 区分费用来源 |
| 用量来源 | 区分 Provider 返回、估算和不可用 |
| 其他计费单位 | 记录 Embedding、Rerank、语音、图像和工具费用 |
| 调用、重试、状态 | 识别放大和失败成本 |
| 排队、首字节、总耗时 | 区分系统等待与模型耗时 |
| 价格、币种、生效日期、来源 | 复现历史费用 |

不要将语音时长、Rerank 文本数、图像尺寸或请求次数伪装成 LLM Token。

## 费用模型

在业务逻辑之外维护价格版本：

```text
总费用 = 输入 Token / 计价单位 × 输入单价
       + 缓存输入 Token / 计价单位 × 缓存输入单价
       + 输出 Token / 计价单位 × 输出单价
       + 推理 Token / 计价单位 × 推理单价
       + 请求费、工具费和其他 Provider 计费项
```

同时记录未知费用和估算费用，不能默认按零计算后宣称降费。

## 调用链盘点

沿入口到 Provider 检查：

- 相同系统规则是否在多个消息或子调用中重复。
- 是否每轮序列化完整历史、用户资料或工具 Schema。
- 工具结果是否未经筛选、截断或摘要直接回灌。
- RAG 是否重复查询、重复片段、Top-K 或 Rerank 范围失控。
- 输出上限是否远高于历史分布，截断是否触发修复调用。
- 并发未命中、断线重连和重试是否执行同一工作。
- 时间戳、字段顺序或动态前缀是否破坏缓存命中。

## 基线流程

1. 定义一个完整业务结果及开始、完成、失败状态。
2. 准备固定合成输入和隔离测试身份。
3. 固定模型、参数、Prompt、知识库和价格版本。
4. 设置测试时间窗、最少请求数和最少完整流程数。
5. 从 Provider 响应或内部用量事件聚合任务级指标。
6. 检查数据完整性，区分返回、估算和不可用值。
7. 保存汇总证据，不保存 Prompt、完整输出或会话标识。

## 热点排序

可使用以下启发式评分：

```text
优先级 = 可验证节省 × 调用频率 × 回滚容易度
       ÷（质量风险 × 实施成本）
```

评分只用于排序。预期节省应以完整流程计算，并包含新增摘要、评分、修复和回退调用。

## 基线门禁

- 请求数必须大于零并达到项目最低要求。
- 已开始、已完成和失败流程必须分别统计。
- 任务级调用总数应与 Provider 或内部事件聚合一致。
- 同一测试的时间窗、模型、参数和价格版本必须固定。
- 无法建立同口径基线时，将工作标记为阻塞，不继续宣称优化收益。
