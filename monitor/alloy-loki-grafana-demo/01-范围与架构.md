# 01：范围与架构

## 当前判断

现有运行栈为 FastAPI、React/Vite 构建后的 Nginx、PostgreSQL + pgvector、Redis 和 Docker Compose 单机部署。后端、Nginx、PostgreSQL、Redis 都可以从容器 `stdout/stderr` 采集日志，因此无需重构业务代码即可落地：

- Grafana Alloy：发现 Docker 容器并采集日志；
- Grafana Loki：单机保存、索引、查询日志；
- Grafana：日志检索和基础面板。

业务 4 个容器加可观测性 3 个容器，共计 7 个。若启用 MinerU，不适用本资源预算，应重新评估 CPU、内存和 GPU 压力。

## 资源预算

| 组件 | CPU 限额建议 | 内存限额建议 | 主要磁盘用途 |
|---|---:|---:|---|
| Alloy | 0.30 核 | 256 MiB | 读取位置、队列与少量状态 |
| Loki | 1.00 核 | 1.5 GiB | 日志块、索引、压缩与保留 |
| Grafana | 0.50 核 | 512 MiB | Dashboard、SQLite、插件与会话 |
| 合计预留 | 约 2 核 | 约 3 GiB | 从 20 GiB 可用空间起步 |

整机最低建议：4 vCPU、8 GiB RAM、40 GiB 可用磁盘；其中为可观测性栈预留约 2 vCPU、3 GiB RAM、20 GiB 磁盘。主机低于 4 vCPU / 8 GiB 或磁盘剩余低于 30 GiB 时，先扩容或降低日志负载。

## 本期边界

实现：容器日志集中采集、7 天 Loki 保留、Grafana 日志搜索和基础面板、基于文本的故障观察、Docker 本地日志轮转、最小安全与隐私保护。

不实现：Prometheus、Tempo、Alertmanager、OpenTelemetry Collector、主机指标、分布式追踪、高可用 Loki、对象存储、Kubernetes、Grafana 公网暴露、业务 API/数据库/业务行为改造。

Alloy + Loki + Grafana 仅是日志方案，不是完整 Metrics + Logs + Traces 平台。

## 架构和隔离

```text
HRagent 业务 Compose（project name = hragent-05）
frontend / backend / postgres / redis
         │ stdout/stderr → Docker json-file logs
         ▼ Docker Socket
独立 observability Compose（project name = hragent-observability）
Alloy ──HTTP Push──▶ Loki ──LogQL Query──▶ Grafana
                       │                    │
                  loki_data            grafana_data
```

必须使用独立 Compose 项目：监控可独立启停升级；停止监控不影响业务；回滚简单；避免生命周期耦合。

## 固定版本

```text
grafana/alloy:v1.17.0
grafana/loki:3.7.3
grafana/grafana:13.0.0
```

不得使用 `latest`。如果拉取失败，只能使用官方仓库中同一大版本的稳定补丁版本，并在交付中说明原因和实际版本。
