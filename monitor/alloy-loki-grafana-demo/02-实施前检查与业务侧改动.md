# 02：实施前检查与业务侧改动

## 先确认实际现状

1. 阅读根目录 `docker-compose.yml`，确认 Compose project name 和四个服务名为 `postgres`、`redis`、`backend`、`frontend`。
2. 搜索现有日志实现及敏感字段：`session_id`、`filename`、`document_id`、`employee_id`、`email`、`token`、`api_key`、`secret`、`password`、`Authorization`、`Cookie`、request/response body。
3. 若实际 project name 不是 `hragent-05`、服务名不一致或日志已有大规模敏感泄露，停止实施，先报告最小修复方案。

建议检索命令：

```bash
rg -n -i 'authorization|cookie|api[_-]?key|secret|password|session_id|filename|request\.body|response\.body' backend
```

## 唯一允许的业务 Compose 改动：日志轮转

在根 `docker-compose.yml` 顶层增加：

```yaml
x-default-logging: &default-logging
  driver: json-file
  options:
    max-size: "20m"
    max-file: "3"
```

并给 `postgres`、`redis`、`backend`、`frontend` 全部增加：

```yaml
logging: *default-logging
```

未来若 MinerU 加入主 Compose，也必须使用同一策略。Loki 7 天保留不会限制 Docker `json-file` 的本地日志文件；未设置轮转会导致磁盘耗尽。

完成后必须通过：

```bash
docker compose -f docker-compose.yml config
```

## 后端日志隐私：完成条件

禁止记录：员工完整姓名、邮箱、手机号、工号、对话全文、上传文档/简历正文、面试回答全文、提示词全文、模型请求/响应全文、Authorization、Cookie、API Key、Client Secret、Password、数据库密码、原始 Access/Refresh Token、请求体和响应体原文。

原始 `session_id` 必须改为不可逆短引用；不得将该引用设置为 Loki 标签：

```python
from hashlib import sha256

def safe_ref(value: str | None) -> str | None:
    if not value:
        return None
    return sha256(value.encode("utf-8")).hexdigest()[:12]

logger.info("documents.text.start | session_ref=%s", safe_ref(session_id))
```

上传文件名不得完整记录；仅可记录扩展名、大小或白名单清洗后的文档类型：

```python
from pathlib import Path

suffix = Path(filename or "").suffix.lower()[:10]
logger.info("documents.text.start | extension=%s", suffix)
```

异常可保留异常类型和堆栈，但不拼接请求/响应正文、不打印环境变量或第三方完整返回，并应脱敏明显凭证。客户端仍只返回通用错误，不能返回堆栈。

## P1（不阻塞 P0）日志标准化建议

P0 采集打通后，推荐增加 `backend/core/logging_config.py`、集中 `dictConfig`、简化 `backend/utils/logger.py`、Request ID 中间件和 `ContextVar`；响应返回 `X-Request-ID`，日志输出单行 JSON。建议字段为 `timestamp`、`level`、`logger`、`message`、`environment`、`service`、`request_id`、`event`、`duration_ms`、`status_code`。

不记录 query string/request body，且 `request_id` 只能是日志 JSON 字段，不能是 Loki 标签。
