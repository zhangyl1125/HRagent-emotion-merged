# 07：测试、部署与回滚

## 实施顺序

1. Settings、超级管理员依赖、迁移/`init_schema` 与权限测试；
2. usage context/repository/tracker、非流式/流式埋点、标准化和估算；
3. trace middleware 与 API request audit；
4. 管理 Analytics、事件、白名单、审计、导出 API；
5. 前端 Tabs、看板、表格、白名单、审计和错误状态；
6. 后端/集成测试、前端 build、Compose、真实或 Mock LLM 核对、停用 Session 与权限回归。

## 必测项目

- 未登录 401；普通用户、非根 admin、认证关闭时均 403；根管理员 role+email 组合 200。
- 根管理员不可停用、删除、移出白名单或被普通管理员重置密码。
- usage 标准化覆盖各 Provider 字段组合、reasoning/cached、缺失、非法负数、字符串和 None。
- 非流式成功/失败/重试/重复 call ID/写库失败隔离；流式多 chunk 仅一条、估算/unavailable、取消、无输出正文。
- API audit 只记录 route template、排除 health/docs、不记录 path ID/query/body、500 仍记录、写库失败不改变响应。
- Bosch 域名边界、停用后 Session 失效、删除后历史仍在、审计无密码、分页排序和时间半开区间。
- `cd frontend && npm run build`，再检查普通用户入口、Tab、空/错误状态、移动表格、根管理员按钮与 CSV。

验收中必须证明：Provider usage total = `llm_usage_events.total_tokens` = 管理页面值；缺失时正确显示 estimated/unavailable。10 万 usage 数据时目标为 overview p95 < 500ms、用户列表 < 700ms、事件页 < 500ms，默认 50 条分页。

## 部署验证

不新增容器，仍使用 frontend/backend/postgres/redis。初始化数据库：`docker compose exec backend python scripts/init_postgres.py`；账号初始化使用现有 provision 脚本。确认根管理员在 `app_users` 为 active admin，且白名单 enabled。构建：`docker compose build backend frontend && docker compose up -d`；健康检查 `curl -k https://localhost:8443/api/v1/health`。登录后 overview/users/whitelist 应为 200，普通用户访问 overview 为 403。

## 回滚

应用层可回滚 AdminPage、从 `main.py` 移除 admin router，并设置 `ADMIN_CONSOLE_ENABLED=false`、`ADMIN_USAGE_TRACKING_ENABLED=false`。默认保留新增事实表以保存审计；若明确需要删除，先备份再删除 `admin_action_audit_log`、`api_request_events`、`llm_usage_events`。Token 统计关闭后 LLM、登录与业务流程必须继续可用，管理页应说明统计已关闭。

## 最终交付

交付完整变更、迁移、文件清单、单元/集成测试、前端 build、Compose 验证、管理员使用说明、回滚、已知限制和 provider/estimated/unavailable 的测试证据。只有全部测试通过、普通用户为 403、流式和非流式都统计、根管理员受保护、管理操作有审计且未记录敏感正文时才可宣布完成。
